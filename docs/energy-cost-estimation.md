# Estimating the Energy Consumed by an LLM Request

## Overview

Xolo estimates the energy consumption of each LLM inference request in **black-box** mode: we don't know the actual datacenter configuration, but we can reason from the model's physical parameters and calibrated assumptions about the type of infrastructure.

The result is always a **range** [min, max] accompanied by a **median value** — not a single supposedly exact number. This honesty about uncertainty is a deliberate choice: the energy consumed by an LLM in production varies by a factor of 5 to 10 depending on real-world conditions.

---

## Algorithm Inputs

| Parameter           | Source                 | Description                                                        |
| ------------------- | ---------------------- | ------------------------------------------------------------------ |
| `activeParams`      | Model configuration    | Number of active parameters per token (e.g., 7 billion for a 7B)   |
| `inputTokens`       | API request            | Tokens sent to the model (prompt)                                  |
| `outputTokens`      | API response           | Tokens generated by the model (completion)                         |
| `tokPerSecLow/High` | Config or heuristic    | Generation throughput in tokens/s (optional)                       |
| `cloudTier`         | Provider configuration | Infrastructure category (hyperscaler, major cloud, small provider) |

---

## Step 1 — Estimating Token Throughput (tokens/s)

The generation throughput (tokens per second) determines the request duration, and therefore the energy consumed using the TDP method (see step 3).

If the administrator hasn't provided a value, an **empirical heuristic** based on model size is used:

```
tps_base = 200 / (bParams ^ 0.4)
```

where `bParams` is the number of billions of active parameters.

Some typical values:

| Model | Active parameters | Estimated tps (base) |
| ----- | ----------------- | -------------------- |
| 7B    | 7 B               | ~100 tok/s           |
| 13B   | 13 B              | ~73 tok/s            |
| 70B   | 70 B              | ~37 tok/s            |
| 400B  | 400 B             | ~15 tok/s            |

To build the uncertainty range:

- **Optimistic scenario**: `tps = tps_base` (recent hardware, good batching)
- **Pessimistic scenario**: `tps = tps_base × 0.5` (saturated hardware, less optimized)

> **Why the 0.4 exponent?** The empirical scaling law states that time per token grows slower than linearly with model size, because larger architectures benefit proportionally more from GPU parallelism.

---

## Step 2 — Computing Request Duration

An LLM request breaks down into two phases with very different properties:

### Prefill Phase (prompt processing)

Prefill processes **all input tokens in parallel** on the GPU. It's a massively parallel operation, analogous to dense matrix multiplication. But the attention mechanism has quadratic complexity O(n²): the longer the context, the more the relative gain from parallelization diminishes.

The speedup factor is therefore **dynamic** depending on prompt length:

```
speedup_prefill = 20 / (1 + inputTokens / 10000)
prefill_duration = inputTokens / (tps × speedup_prefill)
```

Typical values:

| Prompt length | Prefill speedup | Prefill duration (tps=100) |
| ------------- | --------------- | -------------------------- |
| 100 tokens    | ~19.8×          | ~0.05 s                    |
| 1,000 tokens  | ~18.2×          | ~0.55 s                    |
| 5,000 tokens  | ~13.3×          | ~3.8 s                     |
| 20,000 tokens | ~6.7×           | ~30 s                      |
| 50,000 tokens | ~2.9×           | ~172 s                     |

### Decode Phase (response generation)

Decode generates tokens **one by one**, sequentially. This is the time-limiting phase.

```
decode_duration = outputTokens / tps
```

### Total Duration

```
total_duration = prefill_duration + decode_duration
```

**Example** — 7B model, 1000 input tokens, 200 output tokens, tps = 100:

- speedup = 20 / (1 + 1000/10000) = **18.2×**
- Prefill: 1000 / (100 × 18.2) = **0.55 s**
- Decode: 200 / 100 = **2.0 s**
- Total: **2.55 s**

## Step 3 — Method 1: Power-Based Estimation (TDP-based)

This method is inspired by Ji & Jiang (2025): instead of counting operations, it directly models the **electrical power** consumed by GPUs during the request duration.

```
GPU_power = max(MinGPUWatts, WattsPerBParams × bParams)
E_TDP = GPU_power × duration × PUE × 1.20
```

| Term              | Meaning                                                                    |
| ----------------- | -------------------------------------------------------------------------- |
| `MinGPUWatts`     | Minimum GPU power per request (W) — floor related to batching              |
| `WattsPerBParams` | Power in watts per billion active parameters                               |
| `bParams`         | Number of billions of active parameters                                    |
| `duration`        | Total request duration (prefill + decode), in seconds                      |
| `PUE`             | _Power Usage Effectiveness_ — datacenter overhead (cooling, power supply…) |
| `× 1.20`          | +20% for server overheads (CPU, network, KV-cache in HBM memory)           |

**Why a power floor?** Without this floor, a 7B model at a hyperscaler would give `0.10 × 7 = 0.7 W` — unrealistic. In practice, each request is allocated a fraction of a partially loaded GPU, even with aggressive batching. The floor reflects this reality.

**Why +20% and not +10%?** The KV-cache (HBM memory for attention activations) can represent 15–25% of total server power, a component not covered by PUE.

> **Note on double counting**: `WattsPerBParams` implicitly captures part of the GPU memory (HBM) consumption. The +20% includes an approximation of the KV-cache not covered by this term. If `WattsPerBParams` is refined to explicitly include HBM memory, this overhead should be revised downward.

`WattsPerBParams` indirectly encodes hardware efficiency: a recent H100 GPU consumes less per parameter than an A100 or V100.

Parameters by tier:

| Tier           | WattsPerBParams | MinGPUWatts (low/high) | Overhead |
| -------------- | --------------- | ---------------------- | -------- |
| Hyperscaler    | 0.10 – 0.50 W/B | 2 / 20 W               | ×1.20    |
| Major cloud    | 0.30 – 1.00 W/B | 8 / 50 W               | ×1.20    |
| Small provider | 0.50 – 2.00 W/B | 20 / 100 W             | ×1.20    |

## Step 4 — Method 2: Floating-Point Operations Estimation (FLOP-based)

This method counts the number of mathematical operations performed, then converts to energy via the hardware's energy efficiency.

### Computing the Number of FLOP

For a transformer, each token requires approximately `2 × N` floating-point operations, where `N` is the number of active parameters (one multiplication + one addition per parameter).

```
FLOP_per_token = 2 × activeParams

FLOP_prefill = FLOP_per_token × inputTokens
FLOP_decode  = FLOP_per_token × outputTokens
FLOP_total   = FLOP_prefill + FLOP_decode

attentionFactor = 1 + inputTokens / (inputTokens + 5000)
FLOP_total      = FLOP_total × attentionFactor
```

> **Important note**: prefill is counted at **full cost**, identical to decode. GPU parallelization reduces _time_ but not the number of operations performed.

**Attention factor for long contexts**: the attention mechanism has O(n²) complexity in prefill — the longer the prompt, the more attention FLOP represent a significant fraction of the total. The `attentionFactor` approximates this overhead without requiring internal layer parameters:

| Prompt length | attentionFactor | FLOP overhead |
| ------------- | --------------- | ------------- |
| 100 tokens    | 1.02            | +2%           |
| 1,000 tokens  | 1.17            | +17%          |
| 5,000 tokens  | 1.50            | +50%          |
| 20,000 tokens | 1.80            | +80%          |
| 50,000 tokens | 1.91            | +91%          |

_Source: Ji & Jiang (2025); simplified approximation without layer parameters._

### Converting to Energy

```
E_FLOP = FLOP_total / (efficiency × 10⁹) × PUE × 1.20
```

where `efficiency` is in GFLOP/J (gigaflops per joule).

Efficiency ranges by tier:

| Tier                        | Min efficiency | Max efficiency |
| --------------------------- | -------------- | -------------- |
| Hyperscaler (Google, Meta…) | 700 GFLOP/J    | 1000 GFLOP/J   |
| Major cloud (AWS, OVH…)     | 350 GFLOP/J    | 700 GFLOP/J    |
| Small provider              | 150 GFLOP/J    | 350 GFLOP/J    |

> The hyperscaler lower bound is raised to 700 GFLOP/J (was 600): H100/H200 GPUs deployed in 2025 achieve ≥ 700 GFLOP/J effective under typical inference conditions. The upper bound remains at 1000 GFLOP/J — 1200+ GFLOP/J would assume Blackwell hardware with FP8 quantization and perfect batching, too optimistic as a generic value.

## Step 5 — Building the [min, max] Envelope

The two methods each yield a different result. The envelope construction depends on a specific case: whether the **TDP floor is active**.

### Case 1 — TDP Floor Active (small models)

When `WattsPerBParams_low × bParams < MinGPUWatts_low`, the `MinGPUWatts` floor is active in the optimistic scenario. This means the TDP method encodes **infrastructure overhead** (batching, minimal hardware), not the model's physics. In this case, the TDP method is no longer informative and we use **FLOP only**:

```
If WattsPerBParams_low × bParams < MinGPUWatts_low:
    E_low  = FLOP_low
    E_high = FLOP_high
    E_mid  = FLOP_mid  = √(FLOP_low × FLOP_high)
```

### Case 2 — Hybrid (normal-sized models)

When the floor is not active, we take the **conservative envelope** of both methods:

```
E_low  = min(TDP_low,  FLOP_low)
E_high = max(TDP_high, FLOP_high)

TDP_mid  = √(TDP_low  × TDP_high)   ← midpoint TDP method alone
FLOP_mid = √(FLOP_low × FLOP_high)  ← midpoint FLOP method alone

E_mid  = √(TDP_mid × FLOP_mid)      ← geometric mean of per-method midpoints
```

> **Why E_mid from midpoints rather than the envelope?** `√(E_low × E_high)` would be dominated by the widest envelope (often TDP_high), yielding a mid that's too pessimistic. Computing the mid of each method separately, then taking their geometric mean, produces a central value more representative of the consensus between the two approaches.

> **Divergence signal**: if `TDP_mid / FLOP_mid > 5` or `< 0.2`, the two methods diverge significantly. This indicates high uncertainty, often due to poorly calibrated `tokPerSec` or `activeParams` parameters.

- `E_low`: the most favorable scenario retained
- `E_high`: the most unfavorable scenario retained
- `E_mid`: central value, computed as the **geometric mean** (not arithmetic) because the uncertainty is multiplicative — we're more confident about an error factor than an absolute difference

> **Why combine two methods?** The TDP method is more accurate when token throughput is known reliably; the FLOP method is more stable and independent of computation time. Their combination better covers the real uncertainty space. The FLOP-only fallback prevents the infrastructure floor (a constant per tier, independent of the model) from artificially inflating the envelope for small models.

## Infrastructure Presets (CloudTier)

The `WattsPerBParams`, `MinGPUWatts`, and `PUE` parameters depend heavily on the type of datacenter. Xolo offers three levels:

### Hyperscaler (Google, Microsoft, Meta)

- PUE: 1.05 – 1.15 _(highly optimized cooling)_
- GPU utilization: 50 – 80%
- WattsPerBParams: 0.10 – 0.50 W/B params
- MinGPUWatts: 2 – 20 W _(aggressive batching ~100 req/GPU to light ~10 req/GPU)_

### Major Cloud (AWS, OVH, CoreWeave)

- PUE: 1.10 – 1.40
- GPU utilization: 30 – 60%
- WattsPerBParams: 0.30 – 1.00 W/B params
- MinGPUWatts: 8 – 50 W

### Small Provider (startups, regional)

- PUE: 1.20 – 1.60 _(less optimization)_
- GPU utilization: 15 – 40%
- WattsPerBParams: 0.50 – 2.00 W/B params
- MinGPUWatts: 20 – 100 W _(low batching, often dedicated GPU per request)_

> **PUE (Power Usage Effectiveness)**: ratio between the datacenter's total energy and the energy consumed by servers alone. A PUE of 1.0 would be perfect (impossible in practice). A PUE of 1.5 means that for every 1 W consumed by the GPUs, an additional 0.5 W goes to cooling and power distribution.

## Step 6 — Converting to CO₂

Electrical energy doesn't have the same carbon impact depending on where it's produced. Three values are calculated:

```
CO₂ (g) = energy (Wh) × carbon_intensity (gCO₂/Wh)
```

| Scenario    | Carbon intensity | Context               |
| ----------- | ---------------- | --------------------- |
| **France**  | 0.027 gCO₂/Wh    | Nuclear mix           |
| Sweden      | 0.045 gCO₂/Wh    | Hydro/nuclear mix     |
| EU average  | 0.276 gCO₂/Wh    | European mix          |
| World       | 0.475 gCO₂/Wh    | _(default value)_     |
| Natural gas | 0.490 gCO₂/Wh    | Gas-fired power plant |
| Coal        | 0.960 gCO₂/Wh    | _(upper bound)_       |

The display always presents all three values: chosen intensity, France bound (best case), coal bound (worst case).

## Complete Example

**Request**: 7B model, 1000 input tokens, 200 output tokens, hosted at a hyperscaler.

### Step 1 — Throughput

```
tps_base = 200 / 7^0.4 ≈ 100 tok/s
tps_optimistic = 100 tok/s
tps_pessimistic = 50 tok/s
```

### Step 2 — Durations

```
speedup = 20 / (1 + 1000/10000) = 18.18×

Optimistic: prefill = 1000/(100×18.18) = 0.55 s  |  decode = 200/100 = 2.0 s  → total = 2.55 s
Pessimistic: prefill = 1000/(50×18.18) = 1.10 s   |  decode = 200/50  = 4.0 s  → total = 5.10 s
```

_(Note: pessimistic tps = 50 tok/s, so speedup = 20/(1+1000/10000) = 18.18 in both cases — the speedup depends only on prompt length, not on tps.)_

### Step 3 — TDP (hyperscaler preset)

```
GPU_power_low  = max(2 W, 0.10×7) = max(2, 0.7) = 2 W   ← floor active
GPU_power_high = max(20 W, 0.50×7) = max(20, 3.5) = 20 W ← floor active

TDP_low  = 2 W × 2.55 s × 1.05 × 1.20 = 6.4 J   ≈ 1.8 mWh
TDP_high = 20 W × 5.10 s × 1.15 × 1.20 = 141 J   ≈ 39.2 mWh
TDP_mid  = √(6.4 × 141) ≈ 30.0 J ≈ 8.3 mWh
```

### Step 4 — FLOP

```
FLOP_total = 2 × 7×10⁹ × (1000 + 200) = 1.68×10¹³ FLOP

attentionFactor = 1 + 1000/(1000+5000) ≈ 1.167
FLOP_total (adjusted) = 1.68×10¹³ × 1.167 ≈ 1.96×10¹³ FLOP

FLOP_low  = 1.96×10¹³ / (1000×10⁹) × 1.05 × 1.20 ≈ 24.7 J  ≈ 6.9 mWh
FLOP_high = 1.96×10¹³ / (700×10⁹)  × 1.15 × 1.20 ≈ 38.6 J  ≈ 10.7 mWh
FLOP_mid  = √(24.7 × 38.6) ≈ 30.9 J ≈ 8.6 mWh
```

### Step 5 — Envelope and Cross-Method Consistency

```
Floor detection: WattsPerBParams_low × 7 = 0.10 × 7 = 0.7 W < MinGPUWatts_low = 2 W
→ TDP floor active → FLOP-only fallback

E_low  = FLOP_low  = 24.7 J  ≈ 6.9 mWh
E_high = FLOP_high = 38.6 J  ≈ 10.7 mWh
E_mid  = FLOP_mid  = 30.9 J  ≈ 8.6 mWh

Spread: 10.7 / 6.9 ≈ ×1.5  (was ×22 with the raw hybrid envelope)

Ratio TDP_mid / FLOP_mid = 30.0 / 30.9 ≈ 0.97  ← excellent consistency (diagnostic retained)
```

### Step 6 — CO₂ (world mix, 0.475 gCO₂/Wh)

```
CO₂_mid ≈ 8.5 mWh × 0.475 = 4.0 mg CO₂
           (range: 0.049 mg in France → 8.2 mg with coal)
```

## Limitations and Interpretation Caveats

1. **Black-box inference**: we don't know the actual hardware, batching level, or quantization used. Real uncertainty can exceed a factor of 10.

2. **MoE models**: for _Mixture of Experts_ architectures (like Mixtral or GPT-4), `activeParams` must represent the parameters _active per token_, not the total. A model with 56 billion total parameters with 8 experts and top-2 routing has ~14 billion active parameters per token.

3. **Scope limited to inference**: the estimation does not cover training, data storage, or client networking.

4. **Token throughput**: if the provider is configured with `tokPerSecLow/High` values, they replace the heuristic and significantly improve the TDP method's accuracy.

5. **Carbon intensity**: the default value (world mix, 0.475 gCO₂/Wh) is a rough approximation. For a provider hosted in France, the actual impact is 17× lower.
